# Plan: `sensors2mqtt-local-control` — graceful host power control via MQTT

## Goal

Let Home Assistant trigger a **graceful OS shutdown / reboot** of a host
running sensors2mqtt, by exposing per-host **Shutdown** and **Reboot** buttons
over MQTT auto-discovery. The daemon runs *on the host it controls* and calls
`/sbin/shutdown` directly — no SSH, no agent reaching in from outside.

The motivating use case: the Back Shed door automation must power down the two
dashboard Raspberry Pis (`rpiz-dash-1`, `rpiz-dash-2`) **cleanly** before their
mains plugs are cut, so the SD cards are never corrupted by a live power yank.

This daemon is the **control** counterpart to the **`local`** sensor collector,
exactly as **`snmp_control`** is the control counterpart to **`snmp`**.

## Current State

- Each dashboard Pi already runs `sensors2mqtt-local` (`python3-sensors2mqtt`
  0.3.x), enabled, publishing sensors to `ha.welland.mithis.com`.
- Each Pi already has a **dedicated per-host MQTT user** generated by
  gdoc2netcfg in `/etc/sensors2mqtt/env` (e.g. `MQTT_USER=s2m-rpiz_dash_1`).
  The new daemon **reuses that same env file** — no new credentials.
- `host_id()` → `rpiz_dash_1` / `rpiz_dash_2`; the host's HA device already
  exists (`identifiers: [sensors2mqtt_rpiz_dash_1]`). The control buttons attach
  to that **same device**.
- The mains plugs are Tasmota energy-monitoring switches already in HA
  (`switch.au_plug_21` = rpiz-dash-1, `switch.au_plug_13` = rpiz-dash-2,
  `switch.au_plug_12` = speakers).

## Architecture

A new module `src/sensors2mqtt/collector/local_control.py`, a sibling of
`snmp_control.py`, runnable as `python -m sensors2mqtt.collector.local_control`.
It reuses `MqttConfig.from_env()`, `make_client()`, `host_id()`/`host_name()`,
`connection_status_topic()`, and the `discovery` helpers.

| `snmp_control` (`PoeController`) | `local_control` (`PowerController`) |
|---|---|
| reads `snmp.toml`; switches with `write_community` | no config — controls the host it runs on |
| subscribes `…/{node}/port/+/poe/{set,cycle,force/set}` | subscribes `sensors2mqtt/{node_id}/power/{shutdown,reboot}/set` |
| publishes `switch`/`button` discovery per port | publishes two `button` entities for this host |
| `_handle_cycle` → `snmpset` off/on | `_handle_command` → `subprocess.run(["/sbin/shutdown","-h","now"])` / `["/sbin/reboot"]` |
| per-port `busy` lock | single host `busy`/`scheduled` flag |
| connection diag `…/snmp_control/status` | connection diag `…/local_control/status` (Last-Will + heartbeat) |

`node_id = host_id()`, so all topics live under `sensors2mqtt/{node_id}/…`
alongside the host's sensors. The daemon **runs as root** (systemd, no `User=`)
so it can call `/sbin/shutdown` without sudo.

## MQTT topics

| Topic | Dir | Payload | Meaning |
|-------|-----|---------|---------|
| `sensors2mqtt/{node_id}/power/shutdown/set` | in  | `PRESS` | graceful `shutdown -h now` |
| `sensors2mqtt/{node_id}/power/reboot/set`   | in  | `PRESS` | `reboot` |
| `sensors2mqtt/{node_id}/power/state`        | out | `idle` / `shutting_down` / `rebooting` | command acknowledgement |
| `sensors2mqtt/{node_id}/local_control/status` | out | `online` / `offline` | daemon Last-Will + per-cycle heartbeat |

`power/state` is the daemon's **acknowledgement** that it received a command and
is about to halt. It is *not* a confirmation that the host is off — see the
fail-safe section. Published **retained** so HA sees the last action.

## HA auto-discovery entities

Three retained discovery configs, all attached to the existing host device
(identifiers + name only, the safe attach used by `publish_connection_diagnostic`
— never clobbers the manufacturer/model that the `local` collector set):

1. `homeassistant/button/{node_id}/power_shutdown/config` — **"Shutdown"**,
   `command_topic` `…/power/shutdown/set`, `payload_press` `PRESS`,
   `icon: mdi:power`, `entity_category: config` (tucked away so it is not
   fat-fingered).
2. `homeassistant/button/{node_id}/power_reboot/config` — **"Reboot"**,
   `…/power/reboot/set`, `icon: mdi:restart`, `entity_category: config`.
3. The `local_control` connectivity `binary_sensor` (via
   `publish_connection_diagnostic`) — diagnostic; doubles as a coarse liveness
   signal (Last-Will flips it `offline` when the host dies).

Buttons get availability from the daemon's `local_control/status` topic, so a
dead daemon greys them out.

## Why buttons (not a switch)

A switch implies "turn on", but a powered-off Pi cannot receive MQTT to turn
*itself* on — that is the **plug's** job (`switch.au_plug_21`). Responsibilities
split cleanly:

- **plug (Tasmota)** = mains power on/off,
- **local-control** = graceful OS shutdown/reboot.

The door automation presses **Shutdown**, waits for confirmed-off, then cuts the
**plug**.

## CRITICAL fail-safe: never cut power unless confirmed off

> The system must **not** cut power to the RPi hardware unless it is confirmed
> off.

A daemon cannot report "I am fully off" — it is dead by then. So the
authoritative *confirmed-off* signal must come from an **independent observer**
(HA pinging the Pi), not from the daemon. The off-sequence is a state machine:

```
press Shutdown
   ├─ daemon publishes power/state = "shutting_down"   (ack: command received)
   ├─ OS halts → NIC drops → HA ping sensor goes "off"  (CONFIRMED OFF)
   └─ only now: cut the plug
```

Failure handling (fail toward *leaving it powered*):

- **No confirmed-off within the timeout** (ping never drops) → **do NOT cut the
  plug.** Leave mains on, raise a notification/log.
- The 2-minute "cap" from the first draft is **removed** — a cap that cuts
  anyway would violate this rule.

"Confirmed off" = the per-Pi liveness/ping sensor reads `off`. The daemon's
`shutting_down` ack and its `local_control/status` going `offline` are
corroborating signals, not substitutes for the ping drop. This logic lives in
the **HA off-sequence (Phase 2)**; the daemon's job is only to (1) reliably
*trigger* the halt and (2) publish the `shutting_down` ack.

## Daemon behaviour details

- Only act on exact payload `PRESS`; ignore retained/garbage/empty.
- A single `busy` flag (with a lock, like `PortControlState`): once a
  shutdown/reboot is scheduled, ignore further presses so a flood spawns one
  action, not many.
- Dispatch the actual `subprocess.run` from a worker thread (mirrors
  `snmp_control`'s `ThreadPoolExecutor`) so the MQTT loop is never blocked.
- Publish `power/state = shutting_down` **before** invoking `/sbin/shutdown`, so
  the ack is on the wire even though the process is about to be SIGTERM'd by
  systemd during shutdown.
- `--once`: publish discovery + `power/state=idle` + connection online, then
  exit (for testing / discovery refresh), mirroring `snmp_control --once`.
- On clean SIGTERM/SIGINT: publish `local_control/status=offline`, disconnect.

Mirror the `sensors2mqtt-snmp-control` package exactly:

| File | Change |

| `debian/control` | new binary package `sensors2mqtt-local-control` (Depends: `python3-sensors2mqtt`, `python3`, `systemd`) |
| `debian/sensors2mqtt-local-control.install` | `deploy/…service → lib/systemd/system/` |
| `debian/rules` | add `dh_installsystemd -p sensors2mqtt-local-control --no-start` |
| `debian/changelog` | new entry |

Gated by **install scope**: like `snmp-control` is installed only on the
controlling host, `local-control` is installed only where remote power control
is wanted — initially just `rpiz-dash-1` and `rpiz-dash-2`. Fleet-wide rollout
(Ansible role) deferred until proven reliable.

## Tests

`tests/test_local_control.py`, mirroring `tests/test_snmp_control.py`:

- topic parse + dispatch (shutdown vs reboot vs unknown → ignored),
- payload guard (`PRESS` acts; anything else ignored),
- `busy` flag prevents a second concurrent action,
- `_handle_command` invokes the right argv — **`subprocess.run` is mocked; tests
  never actually halt**,
- discovery payload shape (two buttons, correct topics/category/device attach),
- `power/state` ack published before the (mocked) shutdown call,
- `--once` publishes discovery + idle state and returns.

`make lint` (ruff) + `make test` (pytest) green; no regressions in the existing
326 tests.
